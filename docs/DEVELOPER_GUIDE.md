## 1. Project Overview

**DataVault** is a data organization and management web app that lets users store, categorize, search, and visualize their data entries in a sleek, dark-themed dashboard.

### Tech Stack

| Layer        | Technology                                          |
| ------------ | --------------------------------------------------- |
| **Frontend** | React 18, TypeScript 5, Vite 5, Tailwind CSS v3    |
| **UI**       | shadcn/ui (Radix primitives), Framer Motion, Recharts, Lucide icons |
| **Backend**  | Lovable Cloud (Supabase under the hood)             |
| **Database** | PostgreSQL (via Lovable Cloud / Supabase)           |
| **Auth**     | Supabase Auth + Lovable OAuth (Google sign-in)      |
| **Routing**  | React Router DOM v6                                 |
| **State**    | React Query, React Context (AuthContext)            |

### Pages

| Route              | Page        | Purpose                              |
| ------------------ | ----------- | ------------------------------------- |
| `/`                | Home        | Landing page with 3D hero & features |
| `/account`         | Account     | Login / Sign Up / Forgot Password    |
| `/dashboard`       | Dashboard   | Core CRUD, categories, charts, trash |
| `/pricing`         | Pricing     | Plan selection with redirect logic   |
| `/contact`         | Contact     | Support / feedback form              |
| `/reset-password`  | Reset PW    | Password reset flow                  |

## 4. Customization Guide

### 4a) Database Integration

**Current setup:** PostgreSQL via Lovable Cloud (Supabase).

**Where config lives:**
- `src/integrations/supabase/client.ts` — DB client (auto-generated, don't edit)
- `src/integrations/supabase/types.ts` — TypeScript types from DB schema

**Current tables:** `profiles`, `categories`, `entries`

**📁 FILES TO CHANGE to replace with another database (e.g., Firebase, MongoDB):**

1. **Create a new client file** — e.g., `src/integrations/firebase/client.ts`
2. **Update all data-fetching code** in:
   - `src/pages/Dashboard.tsx` — all `supabase.from(...)` calls
   - `src/contexts/AuthContext.tsx` — profile fetching
3. **Replace imports** everywhere that uses:
   ```typescript
   import { supabase } from "@/integrations/supabase/client";
   ```
4. **Update types** to match your new database schema.

> **Tip:** Search the codebase for `supabase` to find every usage point.

---

### 4b) Authentication System

**Current auth method:** Email/password + Google OAuth via Lovable Cloud.

**How it works:**
- `AuthContext.tsx` listens to auth state changes and provides `user`, `session`, `profile` globally.
- `Account.tsx` handles login, signup, and forgot-password forms.
- `ResetPassword.tsx` handles the password reset callback.
- Google sign-in uses `lovable.auth.signInWithOAuth("google")`.

**📁 FILES TO CHANGE to replace auth:**

| File | What to change |
|------|---------------|
| `src/contexts/AuthContext.tsx` | Replace `supabase.auth.*` calls with your auth provider (Firebase Auth, Auth0, etc.) |
| `src/pages/Account.tsx` | Update `handleSubmit` and `handleGoogleSignIn` functions |
| `src/pages/ResetPassword.tsx` | Update password reset logic |
| `src/components/Navbar.tsx` | Update sign-out logic and user display |

**Example: Adding a custom auth provider**

```typescript
// src/contexts/AuthContext.tsx
// Replace:
const { data } = await supabase.auth.signInWithPassword({ email, password });

// With your provider:
const { user } = await myAuthProvider.login(email, password);
```

---

### 4c) API & Backend Logic

**Current setup:** Direct Supabase client calls (no REST API layer). All data operations happen through the Supabase JS SDK.

**Where data operations live:**
- `src/pages/Dashboard.tsx` — CRUD for entries and categories
- `src/contexts/AuthContext.tsx` — User profile fetching

**Adding edge functions (serverless backend):**

Create files in `supabase/functions/your-function-name/index.ts`:

```typescript
// supabase/functions/hello/index.ts
import { serve } from "https://deno.land/std@0.168.0/http/server.ts";

serve(async (req) => {
  return new Response(JSON.stringify({ message: "Hello!" }), {
    headers: { "Content-Type": "application/json" },
  });
});
```

Call from the frontend:

```typescript
const { data } = await supabase.functions.invoke("hello");
```

**📁 FILES TO CHANGE to add new data operations:**

| Action | File |
|--------|------|
| New CRUD operations | `src/pages/Dashboard.tsx` or create a new page |
| New API endpoints | `supabase/functions/<name>/index.ts` |
| New data types | Database migration (auto-updates `types.ts`) |

## 5. Important Notes for Developers

### ✅ Do

- Use **environment variables** for all secrets and API keys
- Use **semantic Tailwind classes** (`bg-primary`, `text-muted-foreground`) — not raw colors
- Keep **components small and focused** — extract logic into custom hooks
- Follow the existing **naming conventions**: PascalCase for components, camelCase for functions
- Use **TypeScript** for all new files
- Add **RLS policies** to any new database tables

### ❌ Don't

- **Never** hardcode API keys or secrets in source code
- **Never** edit auto-generated files (`client.ts`, `types.ts`, `.env`)
- **Never** store user roles on the profiles table (use a separate `user_roles` table)
- **Never** use anonymous signups — always require email verification
- **Never** check admin status via `localStorage` — always use server-side validation

## 6. Common Use Cases

### Replace Lovable Cloud with a custom backend

1. Create your own API server (Express, FastAPI, etc.)
2. Replace all `supabase.from(...)` calls with `fetch()` or Axios calls
3. Update `AuthContext.tsx` with your auth provider
4. Remove `@supabase/supabase-js` from dependencies

**Files to modify:** `Dashboard.tsx`, `AuthContext.tsx`, `Account.tsx`, `ResetPassword.tsx`

---

### Add Google Authentication (already included)

Google OAuth is already configured via Lovable Cloud. The implementation is in:
- `src/pages/Account.tsx` → `handleGoogleSignIn()`
- `src/integrations/lovable/index.ts` → OAuth helper

---

### Connect a new API

1. For **public APIs** — call directly from React components using `fetch()` or React Query
2. For **APIs requiring secrets** — create an edge function:

```
supabase/functions/my-api/index.ts
```

```typescript
serve(async (req) => {
  const apiKey = Deno.env.get("EXTERNAL_API_KEY");
  const response = await fetch("https://api.example.com/data", {
    headers: { Authorization: `Bearer ${apiKey}` },
  });
  return new Response(JSON.stringify(await response.json()));
});
```

---

### Modify UI Design

1. Update CSS variables in `src/index.css` for global theme changes
2. Modify `tailwind.config.ts` for new utilities or colors
3. Edit individual page files in `src/pages/` for layout changes
4. Customize shadcn components in `src/components/ui/` for component-level changes

## 8. Troubleshooting

### Common Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| Blank page after login | Auth redirect mismatch | Check `redirect_uri` in `Account.tsx` matches your domain |
| "No rows returned" error | Missing profile for user | Ensure a trigger creates a profile row on signup |
| Styles not updating | Tailwind purge | Restart dev server; ensure classes aren't dynamically constructed |
| Type errors after DB changes | Stale types | Wait for `types.ts` to auto-regenerate, or re-sync |
| Data not showing | RLS policies blocking | Check that RLS policies allow the authenticated user to read |
| Toast not appearing | Missing `<Toaster />` | Ensure `Toaster` and `Sonner` are in `App.tsx` |

### Debug Tips

1. **Check browser console** for errors — most issues show up here
2. **Network tab** — inspect failed API calls and their response bodies
3. **React DevTools** — inspect component state and context values
4. **Search for `supabase`** — find all backend interaction points
5. **Check RLS policies** — most "empty data" bugs are caused by restrictive policies
